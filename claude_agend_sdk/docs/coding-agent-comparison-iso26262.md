## 1. 핵심 요약

### 1.1 보고서 목적

본 보고서는 ISO 26262 준수가 필수인 자동차 소프트웨어 개발 환경에서 AI 코딩 에이전트를 도입하기 위한 기술적 비교 분석을 제공한다. Stripe의 Minions 프로젝트(2026년 2월 공개)에서 영감을 받은 "one-loop" 자율 코딩 에이전트 패턴을 기반으로, 다음 세 가지 주요 접근법을 심층 평가한다.

- **Claude Code** (hooks + skills + CLAUDE.md 기반 커스터마이징)
- **Goose** (Block의 오픈소스 에이전트 fork 및 커스터마이징)
- **Pi** (최소주의 에이전트) 및 기타 에이전트 (Cline, OpenHands)

### 1.2 핵심 결론

| 항목 | 결론 |
|------|------|
| 1차 추천 | **Claude Code + Hooks + CLAUDE.md** |
| 대안 | Goose fork (멀티 모델, 완전 자체 호스팅 필요 시) |
| 비권장 | Pi (ISO 26262 환경 부적합; 프로토타이핑 전용) |
| 보완 도구 | Cline (ASIL D 극고위험 단위), OpenHands (감사 추적) |

**핵심 통찰**: Stripe가 증명한 바와 같이 "벽이 모델보다 중요하다(The walls matter more than the model)." ISO 26262 준수는 모델의 지능이 아니라 결정론적 게이트, 강제 문서화, 독립적 검토 체계에서 나온다.

### 1.3 경고사항

CodeRabbit 데이터에 따르면 AI 생성 코드는 인간 코드 대비 논리 오류 1.75배, XSS 취약점 2.74배 더 많이 포함한다. AI 코딩 에이전트는 ISO 26262-8 기준 **TCL2 또는 TCL3 도구 자격(Tool Qualification)**을 요구하며, 독립적 인간 검토를 대체할 수 없다.

## 2. 배경 및 동기

### 2.1 Stripe Minions: One-Shot 자율 코딩 에이전트의 실증

2026년 2월, Stripe는 자사의 완전 자율 코딩 에이전트 시스템 **Minions**를 공개했다. 핵심 지표는 다음과 같다.

- **주당 1,300개 이상 PR 병합** (1조 달러+ 결제 코드베이스에서)
- **Slack 메시지 → CI 통과 PR**: 인간 개입 없이 완전 자동화
- **기반 기술**: Block의 오픈소스 에이전트 **Goose** fork

Minions의 아키텍처는 자동차 소프트웨어 개발에 직접적인 시사점을 제공한다.

```
[Slack 태스크] → [Devbox 격리 환경 스핀업 (<10초)] → [Blueprint 실행]
                                                              |
                          ┌───────────────────────────────────┤
                          ↓                                   ↓
               [결정론적 노드]                       [에이전트 노드]
               (git, linting, CI)               (구현, 실패 수정)
                          |                                   |
                          └───────────────────────────────────┘
                                          ↓
                           [최대 2회 CI 재시도 후 에스컬레이션]
                                          ↓
                              [필수 인간 PR 리뷰 게이트]
                                          ↓
                                      [병합]
```

### 2.2 One-Loop 패턴이란?

"One-loop" 패턴은 에이전트가 태스크를 받아 인간 개입 없이 단일 실행 루프 내에서 완료하는 설계를 말한다.

```
┌─────────────────────────────────────────────────────────────────┐
│                        ONE-LOOP AGENT                           │
│                                                                 │
│  입력(태스크) → [LLM 추론] → [도구 실행] → [결과 평가]            │
│                      ↑                          |               │
│                      └──────────────────────────┘               │
│                                 ↓                               │
│                          [완료 판단] → 출력(PR/아티팩트)          │
└─────────────────────────────────────────────────────────────────┘
```

**생산성 레버**: 병렬 실행. 개발자의 주의집중(attention)이 희소 자원이며, 에이전트는 그것을 증폭시킨다.

### 2.3 자동차 소프트웨어에서의 도전

ISO 26262 환경에서 one-loop 에이전트 도입은 다음 긴장 관계를 해결해야 한다.

| 에이전트 특성 | ISO 26262 요건 | 긴장 |
|--------------|---------------|------|
| 자율적 코드 생성 | 독립적 검토 필수 | 에이전트가 검토자가 될 수 없음 |
| 비결정론적 출력 | 재현 가능한 추적성 | 동일 입력 → 다른 코드 가능 |
| 빠른 반복 | 완전한 문서 생성 | 속도와 리거 간 균형 |
| 오류 자가 수정 | 오류 로그 보존 | 수정 전 오류 이력 추적성 |
| 할루시네이션 위험 | MISRA-C 준수 | AI가 존재하지 않는 API를 사용할 수 있음 |

## 3. AI 코딩 에이전트 비교 분석

### 3.1 Claude Code (Anthropic)

#### 3.1.1 아키텍처 개요

Claude Code는 CLI 기반 AI 코딩 에이전트로, 파일 편집, bash 실행, MCP 도구 호출을 핵심으로 한다. ISO 26262 관점에서 가장 중요한 특징은 **완전한 라이프사이클 훅 시스템**이다.

#### 3.1.2 훅 시스템 (Hook System)

Claude Code는 에이전트 실행의 10개 라이프사이클 포인트에서 셸 명령을 삽입할 수 있다.

| 훅 이벤트 | 트리거 시점 | ISO 26262 활용 |
|-----------|------------|----------------|
| `SessionStart` | 새 세션 시작 | 환경 초기화, ASIL 컨텍스트 로드 |
| `UserPromptSubmit` | 사용자 프롬프트 제출 | ISO 26262 문서화 요건 주입, ASIL 컨텍스트 검증 |
| `PreToolUse` | 도구 실행 직전 | 파일 쓰기 전 헤더/문서화 검증, MISRA 체크 강제 |
| `PostToolUse` | 도구 실행 성공 후 | 정적 분석 자동 트리거, 문서 생성 검증 |
| `PostToolUseFailure` | 도구 실행 실패 후 | 실패 로그 기록, 수정 지침 제공 |
| `PermissionRequest` | 권한 요청 발생 | 자동 허용/거부 정책 적용 |
| `Notification` | 알림 발생 | 상태 모니터링 |
| `SubagentStart` | 서브에이전트 시작 | 서브에이전트에 ISO 26262 컨텍스트 주입 |
| `SubagentStop` | 서브에이전트 완료 | 출력 검사, 계속/중단 결정 |
| `Stop` | 메인 에이전트 완료 | ISO 26262 아티팩트 생성 검증, 미완료 시 재개 강제 |

**핵심**: `Stop` 훅에서 미생성 아티팩트 감지 시 에이전트를 강제 재개할 수 있다. 이는 문서화를 "옵션"이 아닌 "완료 조건"으로 만든다.

#### 3.1.3 훅 제어 메커니즘

```jsonc
// PreToolUse 훅 반환 예시: 파일 쓰기 차단
{
  "decision": "deny",
  "reason": "ISO 26262 필수 파일 헤더(ASIL 레벨, 추적성 ID) 누락. 헤더를 추가한 후 다시 쓰십시오."
}

// UserPromptSubmit 훅 반환 예시: 컨텍스트 추가
{
  "additionalContext": "이 프로젝트는 ASIL-B 소프트웨어 컴포넌트입니다. 모든 코드 변경은 SWU 문서(SW Unit Design Doc) 업데이트를 동반해야 하며, MISRA-C:2012 Rule을 준수해야 합니다. 단위 테스트 커버리지는 Branch Coverage 90% 이상이어야 합니다."
}

// Stop 훅 반환 예시: 필수 아티팩트 미생성 시 재개 강제
{
  "continue": true,
  "reason": "단위 테스트 명세서(Unit Test Specification)가 생성되지 않았습니다. docs/unit-test-spec.md를 작성하십시오."
}
```

#### 3.1.4 Skills 시스템

Claude Code의 슬래시 명령(`/skill-name`) 기반 Skills 시스템은 ISO 26262 워크플로우를 캡슐화한다.

- `/iso26262-implement`: 요건 분석 → 설계 → 구현 → 문서화 전 단계 실행
- `/misra-check`: MISRA-C 준수 검사 및 위반 보고서 생성
- `/unit-test-generate`: Branch Coverage 목표치 포함 단위 테스트 자동 생성
- `/traceability-update`: 양방향 추적성 매트릭스 업데이트
- `/asil-assess`: ASIL 레벨 평가 워크시트 생성

oh-my-claudecode(OMC) 프레임워크가 제공하는 오케스트레이션 패턴도 활용 가능하다.

| 패턴 | 설명 | ISO 26262 활용 |
|------|------|----------------|
| `ralph` | 아키텍트 검증 반복 루프 | 설계 문서 완성도 검증 루프 |
| `ultrawork` | 병렬 멀티에이전트 실행 | 독립적 단위 개발 병렬화 |
| `swarm` | N개 에이전트의 태스크 공유 실행 | 대규모 문서화 병렬 생성 |
| `tdd` | 테스트 주도 개발 강제 | ISO 26262 요구 테스트 우선 접근 |

#### 3.1.5 CLAUDE.md / AGENTS.md

CLAUDE.md는 프로젝트 수준의 영속적 지침으로, Claude Code가 매 세션마다 자동으로 읽는다. ISO 26262 요건을 에이전트의 상시 컨텍스트로 인코딩할 수 있다.

```markdown
# CLAUDE.md (ISO 26262 ASIL-B 프로젝트 예시)

## 프로젝트 정보
- 프로젝트명: 전동 파워스티어링 제어 소프트웨어
- ASIL 레벨: ASIL-B
- 코딩 표준: MISRA-C:2012 (필수 규칙 100% 준수, 권고 규칙 95% 이상)

## 필수 준수 사항 (위반 시 코드 생성 중단)
1. 모든 소스 파일 상단에 ISO 26262 파일 헤더 포함 필수
2. 함수 작성 후 반드시 SWU 문서 섹션 업데이트
3. 새 함수 추가 시 단위 테스트 동시 작성 필수
4. Branch Coverage < 90% 시 추가 테스트 케이스 작성
5. 추적성 ID 없는 요건 구현 금지

## ISO 26262 아티팩트 체크리스트 (세션 종료 전 필수 확인)
- [ ] SW Unit Design Doc (docs/swu-design.md) 업데이트
- [ ] Unit Test Specification (docs/unit-test-spec.md) 업데이트
- [ ] MISRA 컴플라이언스 보고서 생성
- [ ] 추적성 매트릭스 (docs/traceability-matrix.md) 업데이트
```

#### 3.1.6 강점 및 약점

**강점**
- 10개 훅 포인트를 통한 세밀한 워크플로우 제어
- 훅을 통해 ISO 26262 문서화를 완료 조건으로 강제 가능
- CLAUDE.md로 프로젝트 표준을 에이전트 컨텍스트에 영속적으로 인코딩
- Skills 시스템으로 ISO 26262 전용 워크플로우 캡슐화
- 성숙한 생태계 및 지속적인 Anthropic 지원

**약점**
- Anthropic 클라우드 의존 (에어갭 환경 불리)
- 코드 특화 (비코드 자동화에 제한적)
- 단일 LLM 제공자 고정

### 3.2 Goose (Block)

#### 3.2.1 아키텍처 개요

Goose는 Block(Square/Cash App)이 개발한 Apache 2.0 오픈소스 AI 에이전트로, 32,000+ GitHub Stars, 408명 기여자를 보유하며 Linux Foundation의 Agentic AI Foundation에 기증되었다.

```
┌─────────────────────────────────────────────────┐
│                   GOOSE AGENT                   │
│                                                 │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │Interface │  │  Agent   │  │  Extensions   │  │
│  │(CLI/GUI) │←→│(Core     │←→│(MCP-based    │  │
│  │          │  │ Logic)   │  │ Tools)        │  │
│  └──────────┘  └──────────┘  └───────────────┘  │
│                                                 │
│  LLM 제공자: Anthropic / OpenAI / Google /       │
│             로컬 모델 (어떤 것이든)               │
└─────────────────────────────────────────────────┘
```

#### 3.2.2 Stripe Minions의 Goose 선택 이유

Stripe가 Goose를 fork하여 Minions의 기반으로 선택한 이유는 다음과 같다.

- **LLM 무관(agnostic)**: 모델 교체 없이 공급자 전환 가능
- **확장성**: MCP 기반 확장으로 ~500개 도구(Toolshed) 연결 가능
- **커스텀 배포(distro)**: 미리 구성된 제공자/확장/브랜딩으로 팀별 표준화
- **Recipe 시스템**: 반복 태스크 워크플로우 자동화
- **Memory MCP**: 글로벌/로컬 메모리로 컨텍스트 지속성

#### 3.2.3 Goose 루프 메커니즘

```
사용자 요청
    ↓
[LLM에 도구 포함하여 전송]
    ↓
[LLM이 도구 호출 결정]
    ↓
[도구 실행]
    ↓
[결과 반환] → [오류 발생 시 오류를 도구 응답으로 캡처 (중단 없음)]
    ↓
[루프 반복 또는 완료]
    ↓
[토큰 최적화: 소형 LLM으로 요약, 무관 내용 삭제]
```

#### 3.2.4 ISO 26262 커스터마이징 방향

Goose는 기본적으로 ISO 26262 게이트를 포함하지 않는다. fork 시 추가해야 할 요소는 다음과 같다.

```
Goose Fork (ISO 26262 Edition)
│
├── 커스텀 Extension: misra-gate
│   └── 파일 쓰기 전 MISRA 체커 자동 실행
│
├── 커스텀 Extension: doc-enforcer
│   └── 태스크 완료 전 ISO 26262 문서 체크리스트 검증
│
├── 커스텀 Recipe: iso26262-implement
│   └── 요건 → 설계 → 구현 → 테스트 → 문서화 순서 강제
│
├── Memory MCP 설정
│   └── ASIL 레벨, 프로젝트 표준, 추적성 ID 영속 저장
│
└── 승인 게이트 Extension
    └── ASIL-C/D 코드 변경 시 인간 승인 요청
```

#### 3.2.5 강점 및 약점

**강점**
- LLM 무관: 에어갭 환경에서 로컬 모델(Llama, Mistral 등) 사용 가능
- 완전 자체 호스팅 가능 (데이터 통제권 완전 유지)
- Apache 2.0: 엔터프라이즈 사용 및 수정에 법적 제약 없음
- Stripe 실사용 검증 (1,300+ PRs/주)
- 멀티 모델 라우팅으로 비용/성능 최적화

**약점**
- ISO 26262 기능을 처음부터 직접 구현해야 함 (높은 초기 비용)
- Claude Code 훅 시스템에 비해 세밀한 라이프사이클 제어 부족
- 자동차 도메인 특화 지원 부재
- 유지보수 비용 (upstream 변경 추적 필요)

### 3.3 Pi (Mario Zechner)

### 3.4 기타 에이전트 (간략 분석)

#### 3.4.1 Cline (Open Source, Apache 2.0)

Cline은 현존 에이전트 중 ISO 26262 맥락에서 **가장 보수적인 설계**를 가진다.

```
Cline 실행 흐름:
계획 수립 → [인간 승인] → 파일 수정 → [인간 승인] → 터미널 실행 → [인간 승인] → ...
```

**장점**:
- 모든 파일 편집 및 터미널 명령에 명시적 인간 승인 필수
- 온프레미스 추론 지원 (에어갭 환경 적합)
- 컴플라이언스 감사 추적 자동 생성
- 팀 사용량 추적

**단점**:
- One-loop 자율 실행 불가 (per-action 승인으로 자동화 제한)
- ASIL-D 극고위험 단위처럼 모든 변경에 인간 눈이 필요한 경우에만 적합

**ISO 26262 추천 시나리오**: ASIL-C/D 단위 구현, 안전 요건 구현 시

#### 3.4.2 OpenHands (Open Source, MIT)

OpenHands는 **이벤트 소싱 아키텍처**로 가장 강력한 감사 추적을 제공한다.

```
OpenHands 아키텍처:
모든 액션 → [불변 EventLog에 기록] → [상태리스 에이전트 처리기]
                    ↓
            [완전 재생 가능한 세션]
```

**장점**:
- 불변 EventLog: ISO 26262 감사 추적에 최적
- `ConfirmationPolicy`: 액션별 승인 여부 구성 가능
- `LLMSecurityAnalyzer`: 모든 도구 호출에 `security_risk` 필드 자동 첨부
- MCP 도구를 SDK 도구로 일급 지원

**ISO 26262 추천 시나리오**: 감사 추적이 핵심인 ASIL 레벨 검증 단계

## 4. ISO 26262 준수 분석

### 4.1 ISO 26262 필수 요건 요약

#### 4.1.1 ASIL 레벨별 요건

ASIL 레벨은 심각도(S0-S3) × 노출도(E0-E4) × 제어가능성(C0-C3)으로 결정된다.

```
ASIL 스펙트럼:
QM → ASIL A → ASIL B → ASIL C → ASIL D
(낮음)                              (높음)
```

**커버리지 요건**:

| 커버리지 | ASIL A | ASIL B | ASIL C | ASIL D |
|----------|--------|--------|--------|--------|
| 구문 커버리지 | ++ (권장) | ++ (권장) | + (추천) | + (추천) |
| 분기 커버리지 | + (추천) | ++ (권장) | ++ (권장) | + (추천) |
| MC/DC 커버리지 | o (선택) | + (추천) | ++ (권장) | ++ (권장) |

#### 4.1.2 필수 산출물 (V-모델, Part 6)

| 산출물 | 해당 조항 | AI 에이전트 생성 가능성 |
|--------|----------|----------------------|
| 소프트웨어 개발 계획 | 6.5 | 보조 생성 (인간 검토 필수) |
| 소프트웨어 안전 요건 명세 (SRS) | 6.6 | 보조 생성 (인간 검토 필수) |
| SW 아키텍처 설계 문서 (SWA) | 6.7 | 보조 생성 |
| SW 단위 설계 문서 (SWU) | 6.8 | **에이전트 자동 생성 적합** |
| 단위 테스트 명세서 | 6.9 | **에이전트 자동 생성 적합** |
| 단위 테스트 보고서 | 6.9 | **에이전트 자동 생성 적합** |
| 통합 테스트 명세서 | 6.10 | 보조 생성 |
| 검증 보고서 | 각 단계 | **에이전트 자동 생성 적합** |
| 양방향 추적성 매트릭스 | 전 단계 | **에이전트 자동 생성 적합** |
| DFA(종속 장애 분석) 보고서 | 6.7 | 인간 주도 (에이전트 지원) |
| 코딩 표준 준수 보고서 (MISRA) | 6.8 | **에이전트 자동 생성 적합** |

#### 4.1.3 검토자 독립성 요건

| ASIL 레벨 | 독립성 수준 | AI 에이전트의 역할 |
|-----------|-----------|------------------|
| ASIL A/B | I1: 다른 사람 | 에이전트 작성 → 다른 사람이 검토 |
| ASIL C | I2: 다른 팀 | 에이전트 작성 → 다른 팀이 검토 |
| ASIL D | I3: 다른 조직 단위 | 에이전트 작성 → 다른 조직 단위가 검토 |

**중요**: AI 에이전트는 자신이 생성한 코드의 독립적 검토자가 될 수 없다. 이는 협상 불가능한 요건이다.

### 4.2 도구 자격 (Tool Qualification, ISO 26262-8)

#### 4.2.1 AI 코딩 에이전트의 TCL 분류

AI 코딩 에이전트는 ISO 26262-8 기준으로 다음과 같이 분류된다.

```
도구 분류:
TI1 (오류 감지 불가) + TD1 (독립 검증 불가) → TCL3 (최고 수준 자격 필요)
TI1 (오류 감지 불가) + TD2 (제한적 독립 검증) → TCL2
TI2 (오류 도입 가능) + TD2/TD3 → TCL2 또는 TCL3

AI 코딩 에이전트 = TI2 (오류 도입 가능) + TD2/TD3
→ TCL2 또는 TCL3
```

#### 4.2.2 자격 방법

| 방법 | 설명 | 주요성 |
|------|------|--------|
| 1a | 현장 사용 이력 | 유용하지만 AI의 비결정론성으로 제한적 |
| 1b | 프로세스 평가 | 에이전트 개발 프로세스 평가 |
| 1c | 검증 테스트 스위트 | **가장 중요**: AI 에이전트별 테스트 케이스 필요 |

#### 4.2.3 AI 에이전트 도구 자격의 핵심 과제

```
비결정론성 문제:
동일 입력 → 다른 출력 → 재현 가능한 검증 어려움
해결책: 특정 모델 버전 고정 + 온도(temperature) 0으로 설정 + 출력 스키마 고정

설명가능성 갭:
AI가 왜 이 코드를 생성했는지 설명 불가
해결책: 에이전트가 생성한 이유 설명 문서를 코드와 함께 저장

할루시네이션 위험:
존재하지 않는 API/함수 사용, 잘못된 표준 참조
해결책: MISRA 게이트, 컴파일 검증 게이트, 정적 분석 게이트

버전 드리프트:
모델 업데이트 후 동일 프롬프트 → 다른 동작
해결책: 모델 버전 고정, 업데이트 전 검증 스위트 재실행
```

#### 4.2.4 ISO/PAS 8800: 신흥 AI 특화 프레임워크

ISO/PAS 8800은 자동차 AI 안전성을 위한 신흥 표준으로, 2025년 기준 자동차 업계 71% 채택률을 보인다. ISO 26262-8 Clause 11을 AI에 적용하는 접근법을 취한다.

**AI 코딩 에이전트 사용 조직은 ISO/PAS 8800 준수 계획을 수립해야 한다.**

## 5. 최종 추천안

### 5.1 1차 추천: Claude Code + Hooks + CLAUDE.md

**Claude Code는 ISO 26262 준수 자동차 소프트웨어 개발에 가장 적합한 AI 코딩 에이전트다.**

#### 5.1.1 선택 이유 (5가지 핵심 논거)

**이유 1: 문서화를 완료 조건으로 강제할 수 있는 유일한 에이전트**

`Stop` 훅을 통해 ISO 26262 필수 아티팩트가 생성되지 않으면 에이전트가 완료를 선언할 수 없다. 이는 "문서화 깜빡음" 문제를 구조적으로 차단한다.

```bash
# Stop 훅 스크립트 예시: 필수 아티팩트 검증
#!/bin/bash
REQUIRED_ARTIFACTS=(
  "docs/swu-design.md"
  "docs/unit-test-spec.md"
  "docs/misra-compliance-report.md"
  "docs/traceability-matrix.md"
)

for artifact in "${REQUIRED_ARTIFACTS[@]}"; do
  if [ ! -f "$artifact" ] || [ ! -s "$artifact" ]; then
    echo '{"continue": true, "reason": "'"$artifact"' 파일이 없거나 비어있습니다. ISO 26262 준수를 위해 이 파일을 생성하십시오."}'
    exit 0
  fi
done

echo '{"continue": false}'
```

**이유 2: 코드 쓰기 전 MISRA 준수 강제**

`PreToolUse` 훅으로 파일 쓰기 전 MISRA 체크를 실행하고, 위반 시 쓰기 자체를 차단한다. 이는 "나중에 고치면 되지"를 방지하는 구조적 해결책이다.

```bash
# PreToolUse 훅: Write 도구 실행 전 MISRA 체크
#!/bin/bash
TOOL_NAME="$1"
FILE_PATH="$2"
CONTENT="$3"

if [[ "$TOOL_NAME" == "Write" && "$FILE_PATH" == *.c ]]; then
  # 임시 파일에 내용 저장 후 MISRA 체커 실행
  TEMP_FILE=$(mktemp --suffix=.c)
  echo "$CONTENT" > "$TEMP_FILE"

  MISRA_RESULT=$(misra-checker "$TEMP_FILE" 2>&1)
  VIOLATIONS=$(echo "$MISRA_RESULT" | grep -c "violation")

  rm "$TEMP_FILE"

  if [ "$VIOLATIONS" -gt 0 ]; then
    echo "{\"decision\": \"deny\", \"reason\": \"MISRA-C 위반 $VIOLATIONS건 발견:\\n$MISRA_RESULT\\n\\n위반 사항을 수정한 후 다시 작성하십시오.\"}"
    exit 0
  fi
fi

echo '{"decision": "allow"}'
```

**이유 3: 세밀한 컨텍스트 주입**

`UserPromptSubmit` 훅으로 매 프롬프트에 ASIL 레벨, 추적성 요건, 테스트 커버리지 목표를 자동 주입한다.

**이유 4: 기존 Anthropic 생태계와의 통합**

Claude Code는 Claude의 코딩 능력을 최대로 활용한다. 자동차 소프트웨어의 복잡한 C/C++ 코드베이스에서 Claude의 코드 이해 능력이 핵심이다.

**이유 5: 성숙한 Skills 생태계**

oh-my-claudecode(OMC) 등의 Skills 프레임워크로 ISO 26262 전용 워크플로우를 즉시 활용 가능하다.

### 5.3 Pi: 비권장 (ISO 26262 환경)

Pi는 ISO 26262 환경에 **권장하지 않는다**.

**Pi가 ISO 26262에 적합하지 않은 구조적 이유**:

1. **안전 게이트 없음**: Full YOLO 기본 설정 — 승인 없이 모든 파일 쓰기 가능
2. **문서화 강제 불가**: 프롬프트 유도만 가능, 구조적 강제 메커니즘 없음
3. **감사 추적 없음**: 생성된 코드의 이유 추적 불가
4. **설계 철학 불일치**: "의견 없는 프리미티브"는 표준 준수에 역행

**Pi가 적합한 경우 (ISO 26262 비적용 영역)**:
- 자동차 프로젝트의 비안전 소프트웨어 (QM 레벨)
- 내부 개발 도구 및 인프라 스크립트
- 프로토타입 및 기술 검증(PoC)
- 레거시 코드베이스 탐색 및 이해
