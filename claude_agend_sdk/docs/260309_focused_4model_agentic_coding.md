# GLM-4.7-Flash vs Qwen3.5-9B vs GPT-4o-mini vs GPT-5-mini
## 에이전틱 코딩 능력 집중 비교

**작성일: 2026년 3월 9일**

---

## 1. 비교 개요 & TL;DR 핵심 결론

본 보고서는 2026년 3월 기준, 소형~중형급 LLM 4종의 **에이전틱 코딩(Agentic Coding)** 능력을 집중 비교한다.
비교 대상은 오픈소스 로컬 실행 모델 2종(GLM-4.7-Flash, Qwen3.5-9B)과 API 기반 상용 모델 2종(GPT-4o-mini, GPT-5-mini)이다.

### TL;DR 핵심 결론

| 항목 | 최강 모델 | 근거 |
|------|----------|------|
| **에이전틱 코딩 종합** | **GLM-4.7-Flash** | SWE-bench Verified 59.2%, LiveCodeBench 64.0 |
| **VRAM 효율 (최소 GPU)** | **Qwen3.5-9B** | Q4_K_M 기준 ~6-7GB, 노트북 8GB GPU 실행 가능 |
| **함수 호출 / 도구 사용** | **Qwen3.5-9B** | BFCL-V4 66.1로 GPT-5-mini(55.5) 능가 |
| **멀티모달 에이전트** | **Qwen3.5-9B** | MMMU-Pro 70.1, OSWorld 41.8, 201개 언어 |
| **API 가성비 (단순 작업)** | **GPT-4o-mini** | $0.15/1M 입력, 가장 저렴한 API |
| **API 추론 능력** | **GPT-5-mini** | AIME 91.1%, GPQA 82.3%, 내장 추론 모드 |
| **수학 추론** | **GLM-4.7-Flash** | AIME 2025: 91.6% (GPT-5-mini의 91.1% 상회) |

**핵심 메시지:**

1. **에이전틱 코딩에서 GLM-4.7-Flash가 압도적이다.** SWE-bench Verified 59.2%는 24GB GPU에서 실행 가능한 모델 중 세계 최고 수준이며, GPT-5-mini의 추정치(30~40%)를 크게 상회한다.
2. **Qwen3.5-9B는 도구 호출에서 GPT-5-mini를 능가한다.** BFCL-V4 66.1로 GPT-5-mini(55.5)와 GPT-4o-mini를 상회하며, 컨텍스트 창 262K(YaRN 확장 시 1M+)는 GPT-4o-mini(128K)의 2배다. 단, SWE-bench 등 순수 코딩 벤치마크는 9B 모델 기준 미공개로 코딩 능력은 미검증 상태다.
3. **GPT-4o-mini는 에이전틱 코딩에 부적합하다.** Aider Polyglot 18.2%로 실제 코딩 에이전트로는 실전 투입이 어렵다.
4. **로컬 실행이 중간 이상 사용량에서 API보다 훨씬 경제적이다.**

### 2.2 오픈소스 모델 VRAM 요구사항

로컬 GPU에서 실행할 때의 VRAM 요구사항을 양자화 수준별로 정리한다.

| 양자화 수준 | GLM-4.7-Flash (30B MoE) | Qwen3.5-9B (9.65B) | 비고 |
|------------|------------------------|---------------------|------|
| **BF16 / FP16** | ~24GB (경계선) | ~19GB (파일 17.92GB) | 고급 GPU 필요 |
| **Q8_0** | ~16GB | ~12GB (파일 9.53GB) | 중급 데스크탑 GPU |
| **Q6_K (권장)** | N/A | ~9GB (파일 7.69GB) | 품질/VRAM 균형 최적 |
| **Q4_K_M** | ~8GB | ~**6-7GB** (파일 5.87GB) | 노트북 8GB GPU 실행 가능 |
| **Q3_K_M** | N/A | ~5.5GB (파일 4.87GB) | 최저 VRAM, 품질 저하 |
| **최소 권장 GPU** | RTX 4070 (Q4) / RTX 4090 (FP16) | RTX 3060 (Q4_K_M) / RTX 4070 (Q6_K) | |

**핵심 차이:**
- **Qwen3.5-9B**: Q4_K_M(파일 5.87GB) 기준 약 6-7GB로, RTX 3060(12GB), 노트북 8GB GPU 등에서 실행 가능하다. Q6_K(~9GB)가 품질과 VRAM의 균형에서 권장 양자화다. BF16은 파일 17.92GB, 실행 VRAM ~19GB.
- **GLM-4.7-Flash**: 총 파라미터 30B이지만 MoE 구조로 활성 파라미터는 3B에 불과하다. Q4_K_M 기준 약 8GB로 RTX 4060(8GB)급에서 실행 가능하며, FP16은 24GB(RTX 3090/4090) 경계선이다.
- **MoE의 장점**: GLM-4.7-Flash는 3B 활성 파라미터만 사용하므로 추론 속도가 빠르고, 30B급 지식을 8GB VRAM으로 활용할 수 있다.

| 가격 항목 | GPT-4o-mini | GPT-5-mini | 비교 |
|----------|-------------|------------|------|
| **입력 가격** (1M 토큰) | $0.15 | $0.25 | GPT-5-mini가 1.67배 |
| **출력 가격** (1M 토큰) | $0.60 | $2.00 | GPT-5-mini가 3.33배 |
| **일일 100K 토큰 비용 (입+출 반반)** | ~$0.0375 | ~$0.1125 | GPT-5-mini가 3배 |
| **월간 비용 (일 100K 기준)** | ~$1.13 | ~$3.38 | |

**로컬 vs API 비용 비교:**

로컬 GPU(RTX 4090 기준 전기료 약 300W, kWh당 $0.12)로 하루 8시간 사용 시:
- 전기료: 약 $0.29/일, 월 $8.64
- 초기 GPU 투자: RTX 4090 약 $1,600 (약 2년 상각)
- 월 상각 비용: ~$67 + 전기료 $8.64 = **~$76/월**

API 비용으로 환산하면, GLM-4.7-Flash를 로컬로 돌리는 것이 월 수백만 토큰 이상 사용 시 API보다 경제적이다. 특히 에이전틱 코딩처럼 토큰 소모가 큰 작업에서는 로컬 실행의 비용 우위가 명확하다.

## 3. 에이전틱 코딩 벤치마크 비교

에이전틱 코딩이란 LLM이 단순 코드 생성을 넘어, **실제 소프트웨어 개발 워크플로**(이슈 분석, 코드 탐색, 수정, 테스트, 디버깅)를 자율적으로 수행하는 능력을 의미한다. 아래 벤치마크별로 4개 모델을 비교한다.

### 3.1 SWE-bench Verified (실제 GitHub 이슈 해결)

SWE-bench Verified는 실제 오픈소스 프로젝트의 GitHub 이슈를 LLM이 자율적으로 해결하는 벤치마크로, 에이전틱 코딩의 **가장 핵심적인 지표**이다.

| 모델 | SWE-bench Verified | 비고 |
|------|-------------------|------|
| **GLM-4.7-Flash** | **59.2%** | 24GB GPU 모델 중 세계 최강 |
| **Qwen3.5-9B** | 미공개 (9B 미발표) | 27B 모델은 72.4이나 9B는 미공개 |
| **GPT-4o-mini** | ~7-10% (추정) | 공식 미발표, 커뮤니티 테스트 기반 추정 |
| **GPT-5-mini** | ~30-40% (추정) | 공식 미발표, 커뮤니티 추정 |

**분석:**
- GLM-4.7-Flash의 59.2%는 경이적인 수치다. 참고로 Claude 3.5 Sonnet이 약 49%, GPT-4o가 약 33.2%였던 시점(2024년 말)을 고려하면, 30B MoE 모델이 이 수준에 도달한 것은 주목할 만하다.
- Qwen3.5-9B는 9B 모델의 SWE-bench 점수가 공개되지 않았다. Qwen3.5-27B는 72.4를 달성했으나 9B 결과는 미발표다. BFCL-V4(66.1), TAU2-Bench(79.1) 등 도구 사용 지표는 뛰어나지만, 순수 코딩(SWE-bench) 능력은 검증되지 않았다.
- GPT-4o-mini는 에이전틱 코딩에서 명확히 약하다. 단순 코드 생성은 가능하지만, 복잡한 리포지토리 수준의 이슈 해결은 어렵다.

### 3.2 LiveCodeBench (경쟁 코딩)

LiveCodeBench는 시간 제한 내 알고리즘 문제 풀이 능력을 측정한다. 에이전틱 코딩과 직접 관련은 적지만, 코드 추론 능력의 지표가 된다.

| 모델 | LiveCodeBench v6 | 비고 |
|------|-----------------|------|
| **GLM-4.7-Flash** | **64.0** (일부 보고: 84.9) | 매우 높은 수준 |
| **Qwen3.5-9B** | **65.6** | GLM-4.7-Flash와 동급, 공식 발표 |
| **GPT-4o-mini** | ~30-35 (추정) | 공식 미발표 |
| **GPT-5-mini** | N/A (미공개) | |

**분석:**
- GLM-4.7-Flash의 64.0(보수적 수치)은 소형 모델로는 매우 높은 수준이다.
- Qwen3.5-9B의 65.6은 GLM-4.7-Flash를 근소하게 앞서는 수치로, 경쟁 코딩 능력에서 양 모델이 동급임을 보여준다.
- 일부 소스에서 GLM-4.7-Flash를 84.9로 보고하는 것은 평가 조건(pass@1 vs pass@5, 문제 세트 버전)의 차이일 수 있다.

### 3.3 HumanEval / MBPP (기본 코딩)

단일 함수 수준의 코드 생성 능력을 측정하는 기본 벤치마크다. 에이전틱 코딩과의 상관성은 낮지만, 기초 코딩 능력의 하한선을 보여준다.

| 모델 | HumanEval | MBPP | 비고 |
|------|-----------|------|------|
| **GLM-4.7-Flash** | N/A (미공개) | N/A | SWE-bench에 집중 발표 |
| **Qwen3.5-9B** | N/A (미공개) | N/A | 멀티모달 벤치마크 집중 발표 |
| **GPT-4o-mini** | **87.2%** | ~85% (추정) | 기본 코딩은 준수 |
| **GPT-5-mini** | ~90%+ (추정) | N/A | 공식 미발표 |

**분석:**
- HumanEval은 2024년 이후 대부분의 모델이 85%+ 달성하여 변별력이 떨어진다.
- GPT-4o-mini가 HumanEval 87.2%를 달성하면서도 SWE-bench에서 7-10%에 머무르는 것은, **단일 함수 생성 능력과 에이전틱 코딩 능력이 전혀 다른 차원**임을 입증한다.
- 에이전틱 코딩은 코드 생성뿐 아니라 리포지토리 탐색, 컨텍스트 이해, 멀티스텝 수정, 테스트 실행 등 복합적 능력이 필요하다.

### 3.4 Aider Polyglot (실제 에이전트 코딩)

Aider Polyglot은 실제 코딩 에이전트(Aider)를 사용하여 다양한 언어의 실제 코딩 작업을 수행하는 벤치마크다. 에이전틱 코딩의 실용적 지표로 중요하다.

| 모델 | Aider Polyglot | 비고 |
|------|---------------|------|
| **GLM-4.7-Flash** | N/A (미공개) | SWE-bench 59.2%로 미루어 높을 것으로 추정 |
| **Qwen3.5-9B** | N/A (미공개) | |
| **GPT-4o-mini** | **18.2%** | 매우 낮음, 에이전틱 코딩에 부적합 |
| **GPT-5-mini** | N/A (미공개) | |

**분석:**
- GPT-4o-mini의 18.2%는 실제 에이전트 코딩 작업에서 5건 중 1건도 제대로 해결하지 못한다는 의미다.
- 이는 GPT-4o-mini를 코딩 에이전트의 백본 모델로 사용하는 것이 **실용적이지 않음**을 명확히 보여준다.
- 반면 GLM-4.7-Flash는 SWE-bench 59.2% 기반으로, Aider Polyglot에서도 상당히 높은 성능이 예상된다.

### 3.5 도구 사용 / 함수 호출 (BFCL, tau2-Bench)

에이전틱 코딩에서 LLM이 외부 도구(파일 시스템, 터미널, API 등)를 정확히 호출하는 능력은 필수적이다.

| 모델 | BFCL-V4 | TAU2-Bench (멀티턴) | OSWorld-Verified (GUI) | 비고 |
|------|---------|---------------------|----------------------|------|
| **GLM-4.7-Flash** | N/A | **79.5** | N/A | 높은 도구 사용 능력 |
| **Qwen3.5-9B** | **66.1** | **79.1** | **41.8** | GPT-5-mini BFCL 능가 |
| **GPT-4o-mini** | N/A | N/A | N/A | 기본 도구 사용 지원 |
| **GPT-5-mini** | **55.5** | N/A | N/A | BFCL-V4 공식 발표 |

**분석:**
- **Qwen3.5-9B의 BFCL-V4 66.1은 GPT-5-mini(55.5)를 10.6점 차로 능가하며, GPT-4o-mini보다도 높다.** 함수 호출/도구 사용 관점에서 4개 모델 중 가장 뛰어난 수치다.
- GLM-4.7-Flash의 TAU2-Bench 79.5와 Qwen3.5-9B의 79.1은 동급 수준으로, 두 오픈소스 모델 모두 멀티턴 에이전트 작업에서 강점을 보인다.
- Qwen3.5-9B의 OSWorld-Verified 41.8은 GUI 에이전트(스크린 캡처 기반 자율 조작) 능력을 보여주며, 멀티모달 에이전트 파이프라인에서의 실용성을 확인시켜 준다.
- GPT-5-mini의 BFCL-V4 55.5는 도구 사용 능력이 있지만, Qwen3.5-9B(66.1)에 크게 뒤처진다.

## 4. 범용 벤치마크 비교 (MMLU, GPQA, AIME)

에이전틱 코딩은 순수 코딩 능력뿐 아니라 추론, 지식, 문제 해결 능력에도 영향을 받는다. 범용 벤치마크를 통해 각 모델의 기초 추론 능력을 비교한다.

### 4.1 종합 지식 (MMLU, MMMLU)

| 모델 | MMLU | MMMLU (다국어) | 비고 |
|------|------|--------------|------|
| **GLM-4.7-Flash** | N/A | N/A | |
| **Qwen3.5-9B** | N/A | **81.2** | gpt-oss-120B의 78.2 능가 |
| **GPT-4o-mini** | **82.0%** | N/A | |
| **GPT-5-mini** | N/A | N/A | |

**분석:**
- Qwen3.5-9B의 MMMLU 81.2는 9.65B 파라미터 모델로서 경이적이다. 120B 모델(gpt-oss-120B)을 능가한다.
- 이는 Qwen3.5-9B가 **다국어 범용 지식**에서 크기 대비 압도적임을 보여준다.
- GPT-4o-mini의 MMLU 82.0%도 준수하나, MMMLU(다국어)에서의 성능은 확인되지 않았다.

- 이는 Qwen3.5-9B의 아키텍처(Gated DeltaNet + sparse MoE 하이브리드)가 추론 효율에서 혁신적임을 시사한다.

## 5. 에이전틱 코딩 유즈케이스별 추천

### 5.1 로컬 실행 (GPU)

로컬 GPU에서 에이전틱 코딩 에이전트를 구동하려는 경우의 추천이다.

#### 시나리오 A: 24GB GPU 보유 (RTX 3090/4090)

| 추천 | 모델 | 이유 |
|------|------|------|
| **1순위** | **GLM-4.7-Flash (FP16)** | SWE-bench 59.2%, 에이전틱 코딩 최강 |
| **2순위** | Qwen3.5-9B (FP16) | 멀티모달 에이전트 파이프라인에 적합 |

- GLM-4.7-Flash를 FP16(~24GB)으로 실행하면 최대 성능을 발휘한다.
- Qwen3.5-9B는 FP16(~20GB)으로도 여유가 있어, 멀티모달 에이전트 작업에 별도 배치할 수 있다.

#### 시나리오 B: 8~12GB GPU 보유 (RTX 4060/3060)

| 추천 | 모델 | 이유 |
|------|------|------|
| **1순위** | **GLM-4.7-Flash (Q4_K_M)** | ~8GB로 에이전틱 코딩 가능 |
| **2순위** | Qwen3.5-9B (Q4_K_M) | ~6GB, 남은 VRAM으로 다른 작업 가능 |

- GLM-4.7-Flash는 Q4_K_M(~8GB)에서도 에이전틱 코딩 능력이 상당하다. MoE 구조 덕분에 양자화 시에도 성능 저하가 상대적으로 적다.
- Qwen3.5-9B는 6GB로 실행 가능하여, 8GB GPU에서 2GB 여유를 남긴다.

#### 시나리오 C: 6GB GPU 보유 (GTX 1660 Super, 노트북 GPU)

| 추천 | 모델 | 이유 |
|------|------|------|
| **유일한 선택** | **Qwen3.5-9B (Q4_K_M)** | ~6GB, 유일하게 실행 가능 |

- 6GB GPU에서는 GLM-4.7-Flash(Q4_K_M ~8GB)를 실행할 수 없다.
- Qwen3.5-9B가 유일한 선택이며, 에이전틱 코딩보다는 멀티모달 에이전트, 범용 대화, 코드 리뷰 등에 활용하는 것이 적합하다.

### 6.3 Qwen3.5-9B에 대한 정확한 포지셔닝

Qwen3.5-9B는 에이전틱 코딩 순위에서 3위이지만, 이는 **이 모델의 실제 가치를 과소평가하는 것**이다. 정확한 포지셔닝은 다음과 같다:

**Qwen3.5-9B는 순수 코딩 특화 모델이 아니라 멀티모달 에이전트 모델이다.**

- **강점**: BFCL-V4 66.1(GPT-5-mini 능가), TAU2-Bench 79.1, 비전-언어 네이티브 통합, 201개 언어 지원, MMMU-Pro 70.1, OmniDocBench 87.7, 컨텍스트 창 262K(YaRN 시 1M+)
- **설계 의도**: 에이전트 워크플로 특화, 함수 호출·멀티모달 입출력을 통한 자율 작업 수행
- **적합한 유즈케이스**:
  - 함수 호출 집약적 에이전트 파이프라인 (BFCL-V4 1위)
  - 스크린샷 기반 GUI 에이전트 (OSWorld-Verified 41.8)
  - 대용량 컨텍스트 처리 (262K, YaRN 시 1M+)
  - 문서/이미지를 포함한 멀티모달 코드 리뷰
  - 다국어 기술 문서 작성 및 번역
  - 저사양 환경(노트북 8GB GPU)에서의 범용 AI 어시스턴트
- **부적합한 유즈케이스**:
  - SWE-bench 수준의 자율적 GitHub 이슈 해결 (9B 기준 미검증)
  - 대규모 리포지토리 수준의 에이전틱 코딩 (GLM-4.7-Flash가 적합)

**핵심 포인트**: Qwen3.5-9B는 LiveCodeBench 65.6, BFCL-V4 66.1(GPT-5-mini 능가), TAU2-Bench 79.1을 달성한 **도구 호출 특화 에이전트 모델**이다. 단, SWE-bench(9B 미공개)와 HumanEval은 미발표로 순수 코딩 능력은 검증되지 않았다. "코딩 에이전트"보다 "도구 호출·멀티모달 에이전트"로 평가해야 한다.

### 6.4 핵심 결론

1. **에이전틱 코딩 = GLM-4.7-Flash**: 24GB 이하 GPU에서 실행 가능한 모델 중 에이전틱 코딩 세계 최강. SWE-bench 59.2%는 GPT-5-mini 추정치(30-40%)를 크게 상회하며, 로컬 실행으로 API 비용도 절감된다.

2. **도구 호출 + 멀티모달 에이전트 = Qwen3.5-9B**: BFCL-V4 66.1로 GPT-5-mini(55.5)를 능가하며 4개 모델 중 함수 호출 최강. 컨텍스트 창 262K(GPT-4o-mini의 2배), OSWorld-Verified 41.8, 비전-언어 통합 추론, 다국어 처리까지 갖춘 에이전트 모델. Q4_K_M 기준 ~6-7GB로 노트북 8GB GPU에서도 실행 가능하다. 단, SWE-bench 9B 결과는 미발표로 순수 코딩 능력은 미검증.

3. **API 추론 = GPT-5-mini**: 내장 추론 모드와 400K 컨텍스트로 분석/계획 단계에 적합하나, 에이전틱 코딩 실행은 로컬 GLM-4.7-Flash에 맡기는 것이 효과적이다.

4. **GPT-4o-mini의 한계**: 에이전틱 코딩에는 부적합(Aider Polyglot 18.2%). 단순 코드 완성, 보조 작업에만 사용하되, 핵심 에이전트 백본으로는 선택하지 말아야 한다.

5. **하이브리드가 정답**: 단일 모델로 모든 작업을 처리하기보다, GPT-5-mini(분석) + GLM-4.7-Flash(실행) + Qwen3.5-9B(멀티모달 검증) 조합이 가장 효과적이다.
