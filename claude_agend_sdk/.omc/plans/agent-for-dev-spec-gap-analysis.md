## 2. 요구사항별 상세 분석

### 2.1 요구사항 1: Human in the Loop

**SPEC 요구사항:**
> "claude code 처럼 중간중간 사람이 필요한 단계에서는 사람과 같이 채팅하여 결정해야함. 신뢰성이 높은 코딩 에이전트를 만들기 위해서는 아직은 필요한 단계로 보임"

**Coverage 평가: 부족**

**보고서에서 다루는 부분:**
- Stripe Minions Blueprint에서 "인간 PR 리뷰" 노드를 결정론적 필수 종단 게이트로 언급 (2.2절, line 140-141)
- ISO 26262 게이트 체계에서 "인간 PR 리뷰 (에이전트 불가)" 단계를 포함 (8.2절, line 1695-1696)
- ASIL 레벨별 리뷰어 자격 요건 언급 (line 1696: "ASIL-A/B: 다른 사람, ASIL-C: 다른 팀, ASIL-D: 다른 조직")

**갭 목록:**
1. **Human in the Loop를 독립 섹션으로 다루지 않음** -- 보고서 전체에 "Human in the Loop"라는 제목의 섹션이 없고, 산발적 언급만 존재
2. **중간 단계 인간 개입 메커니즘 미분석** -- SPEC이 요구하는 것은 PR 리뷰뿐 아니라 "중간중간 사람이 필요한 단계에서 채팅하여 결정"하는 것. 에이전트 실행 중 인간과 대화형으로 결정을 내리는 패턴(예: 설계 결정, 모호한 요구사항 해소, 리스크 판단)이 분석되지 않음
3. **프레임워크별 Human-in-the-Loop 구현 방식 비교 없음** -- Pi SDK의 `input` 이벤트, Claude Code의 대화형 인터페이스, OpenAI Agents SDK의 `handoff` 패턴 등을 Human-in-the-Loop 관점에서 비교하지 않음
4. **인간 개입 시점의 결정 기준 미제시** -- 언제 인간이 개입해야 하는지(confidence threshold, 위험 수준, 불확실성 판단)에 대한 전략이 없음
5. **Claude Code 방식의 대화형 HITL 분석 부재** -- SPEC이 명시적으로 "claude code 처럼"이라고 언급했으므로, Claude Code의 사용자 대화 패턴(tool use 승인, 중간 질문, 계획 확인 등)을 분석해야 하나 이 부분이 빠져 있음

### 2.2 요구사항 2: 결정론적 동작(code) + 비결정론적 동작(agent) 혼합

**SPEC 요구사항:**
> - 방법 A: 결정론적 상태머신의 각 노드를 결정론적 코드 or 비결정론적 one-loop agent로 선택
> - 방법 B: Outer Loop 자체가 one-loop agent (Claude Code 방식), HOOK으로 결정론적 동작 주입
> - 어떤 방법이 customize 가능한지, Claude Code/PI 등에서 어떻게 구현하는지 조사 필요

**Coverage 평가: 충분**

**보고서에서 다루는 부분:**

*방법 A (결정론적 상태머신 + 노드별 선택):*
- Stripe Minions Blueprint 전체 분석 (2절) -- 상태머신의 각 노드가 결정론적(사각형) 또는 에이전트(구름)로 구분되는 패턴을 상세히 설명
- Pi SDK `setActiveTools()` + 상태머신 컨트롤러로 Phase별 도구 제한 구현 (4.3.5절, 8.1절)
- LangGraph를 활용한 상태머신 구현 예제 (6.5절) -- `add_node("hydrate_context", ...)` (결정론적) vs `add_node("implement", ...)` (비결정론적) 패턴
- Pi SDK 계획 모드 상태머신 예제 -- 3-상태 상태머신 검증된 패턴 (4.4절)

*방법 B (One-loop agent + HOOK으로 결정론적 주입):*
- Claude Code Hook 시스템 전체 분석 (5절) -- 17개 이벤트로 결정론적 동작을 주입
- Pi SDK 28개 이벤트 시스템 전체 분석 (4절) -- `context`, `tool_call`, `tool_result` 등으로 결정론적 벽 구축
- OpenAI Agents SDK Guardrails 분석 (6.2절) -- one-loop agent에 결정론적 가드레일 부착

*프레임워크별 커스터마이징 비교:*
- 6가지 핵심 능력(C1-C6) 기준 3개 프레임워크 비교 (3절)
- 5가지 구체적 시나리오별 구현 비교 (7절)
- 종합 비교 매트릭스 (7.6절)
- 최종 권고사항 (9절) -- Pi SDK > Claude Code > OpenAI Agents SDK 순위 및 근거

**근거:** 보고서의 핵심 주제가 바로 이 요구사항이며, 방법 A와 방법 B 모두를 코드 예제와 함께 심층적으로 분석하고 있다. Stripe Minions 참조도 SPEC에서 제시한 두 블로그 포스트를 반영한 것이다.

### 2.3 요구사항 3: 신뢰성과 효율성 trade-off

**SPEC 요구사항:**
> "모든 Agent를 langgraph방식의 상태머신으로 만든다면 신뢰성을 올라가지만, 해당 시스템이 처리할 수 있는 task가 적어지므로 효율성은 떨어짐. 이를 고려해서 둘다 챙긴 시스템을 만들어서 회사내부에 정착시키는것이 목표임."

**Coverage 평가: 부족**

**보고서에서 다루는 부분:**
- Stripe Minions의 "벽이 모델보다 중요하다" 철학 (2.3절) -- 신뢰성 관점의 통찰
- Pi SDK의 인프로세스 실행 vs Claude Code 서브프로세스 오버헤드 비교 (3.3절) -- 효율성 관점의 간접적 언급
- 이벤트 수가 더 많으면 개입 기회가 더 많다는 논리 (3.2절) -- 신뢰성 관점
- "모든 에이전트를 LangGraph 상태머신으로" 만드는 것의 학습 비용 언급 (6.5절 마지막 줄)

**갭 목록:**
1. **신뢰성-효율성 trade-off를 독립 주제로 분석하지 않음** -- 보고서는 주로 "신뢰성을 어떻게 높일 것인가"에 집중하며, 효율성과의 trade-off를 체계적으로 다루지 않음
2. **"효율성"의 정의 부재** -- SPEC이 말하는 효율성이 (a) 처리 가능한 task의 범위(유연성), (b) 개발 속도(time-to-market), (c) 런타임 성능(오버헤드), (d) 에이전트 자율성 중 어떤 것인지 명확히 하고 각각을 분석해야 하나 이 분석이 없음
3. **"둘 다 챙긴 시스템" 설계 전략 미제시** -- 보고서의 아키텍처 설계(8절)는 Pi SDK 기반 결정론적 게이트 체계에 집중되어 있으며, "언제 결정론적 제어를 적용하고 언제 에이전트에 자유를 줄 것인가"에 대한 스펙트럼/전략이 없음
4. **태스크 유형별 결정론/비결정론 배분 전략 없음** -- 예: 단순 버그 수정은 one-loop agent + 최소 게이트, 신규 기능 개발은 상태머신 + 풀 게이트 등. SPEC이 원하는 "둘 다 챙긴 시스템"의 핵심인 적응적 전략이 부재
5. **정량적 비교 부재** -- 상태머신 방식 vs hook 방식의 개발 비용, 유지보수 비용, 런타임 오버헤드, 처리 가능 태스크 범위 등의 정량적/반정량적 비교가 없음
6. **"회사 내부 정착" 관점 부재** -- 팀 온보딩, 학습 곡선, 점진적 도입 전략, 기존 워크플로우와의 통합 등 조직 차원의 고려사항이 다뤄지지 않음
7. **결정론적 제어의 강도를 조절하는 메커니즘 미분석** -- 프로젝트 위험도, ASIL 레벨, 코드 성숙도 등에 따라 게이트 강도를 동적으로 조절하는 전략이 없음

### 작업 2: 신뢰성-효율성 trade-off 분석 및 보고서 섹션 추가

**목적:** SPEC 요구사항 3의 갭 해소

**작업 내용:**
1. "효율성" 개념 정의 및 다차원 분석
   - 처리 가능 태스크 범위 (유연성)
   - 개발/설정 비용 (time-to-deploy)
   - 런타임 오버헤드 (게이트 실행 시간)
   - 에이전트 자율성 (창의적 문제 해결 범위)
2. 태스크 유형별 결정론/비결정론 배분 전략 설계
   - 단순 버그 수정: minimal gates (hook 방식, 방법 B)
   - 기능 개발: moderate gates (상태머신 + hook 혼합)
   - 안전 관련 코드: full gates (엄격한 상태머신, 방법 A)
   - 탐색적 프로토타이핑: no gates (순수 one-loop agent)
3. 결정론적 제어 강도의 스펙트럼 정의
   - Level 0: 순수 one-loop agent (게이트 없음)
   - Level 1: 최소 게이트 (커밋 전 린트/테스트만)
   - Level 2: 중간 게이트 (상태별 도구 제한 + 컨텍스트 주입)
   - Level 3: 전체 게이트 (Stripe Minions 패턴 풀 적용)
   - Level 4: 최대 게이트 (ISO 26262 ASIL-D 수준)
4. "둘 다 챙긴 시스템" 아키텍처 설계
   - 프로젝트/태스크 메타데이터 기반 게이트 강도 자동 결정
   - 설정 파일로 게이트 프로필 관리 (`.agent-profile.json`)
   - 점진적 도입 전략 (Level 0에서 시작, 점진적으로 Level 올림)
5. 조직 정착 전략
   - 팀 온보딩 계획
   - 기존 CI/CD 파이프라인과의 통합
   - 성공 메트릭 정의

**산출물:** `reliable-dev-agent-deterministic-mixing.md`에 "신뢰성-효율성 trade-off 전략" 섹션 추가, 또는 별도 보고서

**수용 기준:**
- [ ] "효율성"의 다차원 정의가 명시됨
- [ ] 태스크 유형별 결정론/비결정론 배분 전략표 포함
- [ ] 결정론적 제어 강도 스펙트럼(Level 0-4) 정의 및 각 레벨별 구현 방법 포함
- [ ] "언제 어떤 레벨의 게이트를 적용할 것인가"에 대한 의사결정 플로우차트 포함

### 요구사항 1이 "부족"인 이유

- "Human in the Loop"이라는 용어가 보고서 본문에 등장하지 않음
- 인간 개입은 오직 "인간 PR 리뷰"라는 최종 게이트로만 다뤄짐
- SPEC이 요구하는 "중간중간 사람과 채팅하여 결정"하는 패턴이 분석되지 않음
- 프레임워크별 대화형 인간 개입 메커니즘 비교가 없음

### 요구사항 3이 "부족"인 이유

- 보고서는 "신뢰성을 어떻게 극대화할 것인가"에 집중하며, 신뢰성과 효율성의 균형점을 다루지 않음
- "모든 에이전트를 상태머신으로 만들면 효율성이 떨어진다"는 SPEC의 핵심 우려에 대한 직접적 응답이 없음
- 태스크 유형/위험도에 따라 결정론적 제어 수준을 조절하는 적응적 전략이 부재
- 회사 내부 정착이라는 조직 관점이 전혀 반영되지 않음
